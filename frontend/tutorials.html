    Getting Started

    You can freely explore the entire platform without needing to register or log in. The Model Catalog is accessible to all visitors, allowing you to browse a wide range of Earth Observation (EO) machine learning models. You can view detailed information about each model, including metadata, supported tasks, and download links for sample datasets.

    To upload your own models and contribute to the community, you need to create an account. Register via the Registration page and log in to access the upload functionality. After logging in, you can also track your uploaded models and manage your contributions through your personalized Dashboard.

    About the Website

    The HuggingEarth platform serves as a centralized hub for discovering, sharing, and using machine learning models designed for geospatial and EO data. It enables users to find models for tasks like classification, segmentation, and prediction, streamlining workflows for research, monitoring, and data analysis.

    What is STAC?

    The SpatioTemporal Asset Catalog (STAC) is a standardized format for describing geospatial assets, ensuring consistency and interoperability between datasets. Our models and metadata comply with the STAC specification, allowing seamless integration with geospatial tools and libraries like pystac.

    STAC Files Structure and Requirements

    This section describes the required and optional fields in the STAC Collection and Item files, including the validation rules and allowed values.

    Collection Structure

    A Collection in the SpatioTemporal Asset Catalog (STAC) represents a logical grouping of related geospatial assets and their shared metadata. It acts as a higher-level container that provides a summary of the entire dataset, including information such as the dataset’s spatial and temporal extent, licensing details, keywords, and the contributing providers. Collections often represent a dataset, such as satellite imagery over a region or machine learning-ready training data for environmental monitoring. Collections may contain one or more Items.

    Compatible to: STAC Collection Specification

    Field Description Required Notes
    type Must be set to "Collection" to validate as a STAC Collection. Yes Fixed value "Collection".
    stac_version The STAC version this collection implements. Yes
    stac_extensions List of extension identifiers implemented by the collection. No Must be an array of strings.
    id Unique identifier for the collection. Yes Must be unique across all collections.
    title A short descriptive one-line title for the Collection. No
    description Detailed description of the collection. Yes Supports CommonMark syntax for rich text.
    keywords List of keywords describing the collection. No Optional, array of descriptive keywords.
    license License of the collection's data. Yes Must be a valid SPDX License identifier:
    • "CC-BY-4.0" - Creative Commons Attribution 4.0 International
    • "CC-BY-SA-4.0" - Creative Commons Attribution-ShareAlike 4.0 International
    • "MIT" - MIT License
    • "Apache-2.0" - Apache License 2.0
    • "GPL-3.0-or-later" - GNU General Public License 3.0 or later
    • "BSD-3-Clause" - BSD 3-Clause License
    • "CC0-1.0" - Creative Commons Zero v1.0 Universal
    • "EUPL-1.2" - European Union Public License v1.2
    • "other" - Use this if none of the official SPDX licenses apply. Consider adding a link to the custom license text or a description for clarity.
    If "other" is used, a warning will appear recommending additional details for transparency.
    providers List of organizations contributing to the data, such as those capturing, processing, or hosting the data. No Must be an array of provider objects. Each provider must follow these rules:
    • name (required): The organization's name. Must be a non-empty string.
    • description (optional): A description of the provider. Must be a string if provided.
    • roles (optional): An array of roles. Valid roles are:
      • licensor: The entity responsible for licensing the data.
      • producer: The organization that produced the original data.
      • processor: The organization that processed or modified the data.
      • host: The entity hosting the data. There must be no more than one host role, and it must be the last entry in the list if present.
    • url (optional): A URL pointing to the provider's homepage or additional information. Must be a valid URL starting with http:// or https://.
    If the providers array is not null, all entries must follow the specified format.
    extent Spatial and temporal extents of the collection, defining the geographical bounds and time periods covered. Yes Must be an object containing the following properties:
    • spatial.bbox (required): An array of bounding boxes, where each bounding box is an array of 4 or 6 numbers representing:
      • 2D Bounding Box: [minLon, minLat, maxLon, maxLat]
      • 3D Bounding Box: [minLon, minLat, minElevation, maxLon, maxLat, maxElevation]
      Validation rules:
      • Coordinates must adhere to WGS 84 bounds:
        • Longitude (minLon, maxLon) must be between -180 and 180.
        • Latitude (minLat, maxLat) must be between -90 and 90.
      • If 3D, the elevation values (minElevation, maxElevation) must be valid, and minElevation must be less than maxElevation.
    • temporal.interval (required): An array of time intervals, where each interval is an array of two elements [start, end]:
      • start: The start time of the interval in RFC 3339 format (e.g., 2023-01-01T00:00:00Z) or null.
      • end: The end time of the interval in RFC 3339 format or null.
      Validation rules:
      • Both start and end must be valid timestamps or null.
      • If both start and end are provided, start must be less than or equal to end.
    This field is mandatory for all collections and must follow the specified format.
    summaries

    A map of property summaries, providing either a set of values, a range of values, or a JSON Schema.

    This field is strongly recommended but optional and can include:

    • Range Summary: Specifies a minimum and maximum value for numeric or string properties.
      • minimum and maximum must be numbers or strings.
      • minimum must not exceed maximum.
    • JSON Schema Summary: A JSON Schema that defines the expected structure of the property values.
      • Must be a valid object containing the $schema keyword.
      • The $schema keyword should point to a recognized schema version (e.g., http://json-schema.org/draft-07/schema#).
      • A warning will be issued if a non-standard schema version is used.
    • Value Set: Can include specific values (e.g., an array of possible values for categorical properties).
    		No
    • Must be an object if provided, not an array.
    • Each key in the map represents a property, and its value defines the summary.
    • Examples:
      • { "datetime": { "minimum": "2022-01-01T00:00:00Z", "maximum": "2022-12-31T23:59:59Z" } } (Range summary)
      • { "classification": { "$schema": "http://json-schema.org/draft-07/schema#", "type": "string" } } (JSON Schema summary)
      • { "bands": ["B1", "B2", "NDVI"] } (Value set summary)
    links

    A list of references to related documents or resources.

    This field is required and each link must be a valid object that may contain the following keys:

    • href (Required): The URL or path to the referenced document. Must be a non-empty string.
    • rel (Required): The relation type (e.g., self, item). Must be a non-empty string.
    • type (Optional): The media type of the referenced document (e.g., application/json). Must be a string if provided.
    • title (Optional): A human-readable title for the link. Must be a string if provided.
    • method (Optional): The HTTP method (e.g., GET, POST). Must be one of GET, POST, PUT, DELETE, PATCH, OPTIONS, or HEAD if provided.
    • headers (Optional): A set of HTTP headers for the request. Must be an object, where each key is a header name and the value is a string or an array of strings.
    • body (Optional): The body of the request. Must be an object if provided.
    		 Yes
    • The links field must be an array of link objects.
    • Each link object must contain the href and rel keys as non-empty strings.
    • The type, title, method, headers, and body keys are optional.
    • Valid HTTP methods for method are GET, POST, PUT, DELETE, PATCH, OPTIONS, and HEAD.
    • Each header value in headers must be a string or an array of strings.
    • Examples:
      • { "href": "https://example.com/resource", "rel": "self" }
      • { "href": "https://example.com/resource", "rel": "item", "method": "GET", "headers": { "Accept": "application/json" } }
    item_assets

    A dictionary describing the available assets in the member items.

    This field is optional but must conform to the following structure if provided:

    • Key: Must be a non-empty string representing the asset key.
    • Value (Item Asset Object): Must be a valid object containing:
      • title (Optional): Human-readable name of the asset. Must be a string if provided.
      • description (Optional): A detailed description of the asset. Must be a string if provided.
      • type (Optional): Media type of the asset (e.g., image/png, application/json). Must be a string if provided.
      • roles (Optional): Array of roles describing the use of the asset (e.g., thumbnail, data). Each role must be a string.
    		 No
    • The item_assets field must be an object (dictionary).
    • Each key must be a non-empty string, representing the name of the asset.
    • Each asset must contain at least one of the following fields: title or type.
    • The roles field must be an array of strings if provided.
    • Examples:
      • { "thumbnail": { "title": "Preview Image", "type": "image/png", "roles": ["thumbnail"] } }
      • { "metadata": { "title": "Metadata File", "type": "application/json" } }
    • If roles is provided, each role must be a string (e.g., "data", "overview").
    Item Structure

    An Item in STAC represents a structured metadata record that describes a single resource or asset, such as a machine learning model. It provides key information about the resource, including its purpose, relevant files, and how it can be accessed. This allows users to easily identify and utilize specific models or data assets within a catalog.

    Compatible to: STAC Item Specification

    Field Description Required Notes
    type Must be set to "Feature" to validate as a STAC Item. Yes Fixed value "Feature".
    stac_version The STAC version this item implements. Yes
    stac_extensions List of extension identifiers implemented by the item. No Must be an array of strings.
    id Unique identifier for the item within its collection. Yes Must be unique within the collection.
    collection The ID of the STAC Collection this Item references. This field is required if a link with a "collection" relation type is present; otherwise, it must be null. No
    • Must reference a valid id of a Collection.
    • Uploading items is currently only possible if the referenced collection is already uploaded.
    geometry

    The full footprint of the asset represented by this item, formatted as a GeoJSON object. Must follow the RFC 7946, section 3.1 format if a geometry is provided, or section 3.2 if no geometry is provided.

    		Yes, if bbox is provided
    • Must conform to GeoJSON standards.
    • Supported geometry types: Point, MultiPoint, LineString, MultiLineString, Polygon, MultiPolygon, and GeometryCollection.
    • If null, the item's bounding box (bbox) must still be valid.
    bbox

    The bounding box of the asset represented by this item. Must be formatted according to RFC 7946, section 5.

    		Yes, if geometry is provided
    • Must be an array with exactly 4 or 6 numeric elements.
    • Format for 2D bounding box: [minLon, minLat, maxLon, maxLat].
    • Format for 3D bounding box: [minLon, minLat, minElevation, maxLon, maxLat, maxElevation].
    • Longitude (minLon, maxLon) must be between -180 and 180.
    • Latitude (minLat, maxLat) must be between -90 and 90.
    • Elevation (minElevation, maxElevation) must be valid if provided, and minElevation must not exceed maxElevation.
    • This field is required if geometry is not null and prohibited if geometry is null.
    properties

    A dictionary of additional metadata for the Item. This field must include a valid datetime field formatted according to RFC 3339 or be null.

    Additionally, this field includes machine learning model-specific metadata described below:

    • mlm:name (Required): Name of the machine learning model. Must be a non-empty string.
    • mlm:architecture (Required): The architecture of the model (e.g., ResNet, RandomForestClassifier).
    • mlm:tasks (Required): List of tasks (e.g., "segmentation", "classification").
    • mlm:framework (Optional): Framework used for training (e.g., "PyTorch", "scikit-learn").
    • mlm:framework_version (Optional): Version of the framework used.
    • mlm:memory_size (Optional): Memory size required during inference, in bytes. Must be a positive integer.
    • mlm:total_parameters (Optional): Total number of parameters (trainable + non-trainable).
    • mlm:pretrained (Optional): Boolean indicating if the model is pretrained.
    • mlm:pretrained_source (Required if mlm:pretrained is true): The dataset or description of the pretraining source.
    • mlm:batch_size_suggestion (Optional): Suggested batch size for inference. Must be a positive integer.
    • mlm:accelerator (Optional): Specifies the intended hardware accelerator (e.g., "cuda", "amd64").
    • mlm:accelerator_constrained (Optional): Boolean indicating if the model can only run on the specified accelerator. Defaults to false.
    • mlm:accelerator_summary (Optional): Description of the required accelerator (e.g., GPU generation).
    • mlm:accelerator_count (Optional): Minimum number of accelerator instances required.
    • mlm:input (Required): Description of the model input:
      • name: Describes the input data source (e.g., "Sentinel-2 EO Data").
      • bands: Array of bands (e.g., "B01", "B02").
      • shape: Array describing input dimensions (e.g., [-1, 12, 512, 512]).
      • dim_order: Array defining the order of dimensions (e.g., ["batch", "channel", "height", "width"]).
      • data_type: Data type (e.g., "float32").
    • mlm:output (Required): Description of the model output:
      • name: Describes the output (e.g., "Segmentation Map").
      • tasks: List of output-related tasks (e.g., "semantic-segmentation").
      • result.shape: Array describing the output shape (e.g., [-1, 512, 512]).
      • result.dim_order: Order of output dimensions (e.g., ["batch", "height", "width"]).
      • result.data_type: Output data type (e.g., "uint8").
    • mlm:hyperparameters (Optional): JSON object with training or inference hyperparameters (e.g., {"learning_rate": 0.001}).
    		 Yes
    • Must be a non-empty JSON object.
    • If datetime is not provided or invalid, it will be set to null.
    • Format for datetime: YYYY-MM-DDTHH:MM:SSZ (e.g., 2023-01-01T00:00:00Z).
    • If datetime is null, the following fields must be provided:
      • start_datetime: The start time in RFC 3339 format or null.
      • end_datetime: The end time in RFC 3339 format or null.
    • If both start_datetime and end_datetime are provided, start_datetime must be less than or equal to end_datetime.
    • Examples:
      • { "datetime": "2023-01-01T00:00:00Z" } (Valid single timestamp)
      • { "start_datetime": "2023-01-01T00:00:00Z", "end_datetime": "2023-12-31T23:59:59Z" } (Valid time range)
      • { "datetime": null, "start_datetime": null, "end_datetime": null } (No specific time period)
    • If start_datetime or end_datetime are invalid, they will be set to null.
    links

    A list of link objects pointing to related resources and URLs. This field is required and must be an array of valid link objects.

    Each link object must follow these rules:

    • href (Required): The URL or path to the resource. Must be a non-empty string.
    • rel (Required): The relation type (e.g., self, item, collection). Must be a non-empty string.
    • type (Optional): The media type of the resource (e.g., application/json). Must be a string if provided.
    • title (Optional): A human-readable title for the link. Must be a string if provided.
    • method (Optional): The HTTP method to be used (e.g., GET, POST). Must be one of:
      • GET
      • POST
      • PUT
      • DELETE
      • PATCH
      • OPTIONS
      • HEAD
    • headers (Optional): HTTP headers for the request. Must be an object where each key is a header name and the value is a string or an array of strings.
    • body (Optional): The request body. Must be a valid object if provided.
    		 Yes
    • The links field must be an array of link objects.
    • Each link object must contain href and rel as non-empty strings.
    • The type, title, method, headers, and body fields are optional.
    • Valid HTTP methods for method are GET, POST, PUT, DELETE, PATCH, OPTIONS, and HEAD.
    • Each header value in headers must be a string or an array of strings.

    Examples:

    • { "href": "https://example.com/resource", "rel": "self" } (Basic example)
    • { "href": "https://example.com/resource", "rel": "item", "method": "GET", "headers": { "Accept": "application/json" } } (With method and headers)
    assets

    The assets field provides metadata and access information for downloadable or streamable data associated with an Item. It contains a dictionary where the keys represent asset names and the values describe the assets themselves.

    Each asset entry must follow these rules:

    • href (Required): URI to the asset object. Must start with http://, https://, or s3://. Trailing slashes are significant.
    • title (Optional): A user-friendly title for the asset. Must be a string.
    • description (Optional): Additional details about the asset, such as processing information. CommonMark 0.29 syntax may be used for rich text. Must be a string.
    • type (Optional): Media type of the asset (e.g., application/json, image/png). Commonly used media types include:
      • image/tiff; application=geotiff
      • image/jpeg
      • application/geo+json
      • application/vnd.laszip+copc
      • text/plain
      If the type is not a common STAC media type, a warning may be issued.
    • roles (Optional): An array of semantic roles that describe the asset (e.g., thumbnail, overview, data). Each role must be a string.
    • Links to Models, Collections, or Items: Depending on the type of resource:
      • item_id (Optional): A reference to the related STAC Item's ID. The onDelete behavior for this foreign key is set to CASCADE, meaning associated assets will be deleted if the related Item is deleted.
      • model_id (Optional, commented in code): Can reference an ML model resource (if present).
      • collection_id (Optional, commented in code): Can reference a STAC Collection (if needed).
    		 Yes

    The assets field must be a dictionary where keys represent asset names, and values describe the assets.

    Examples:

    • { "thumbnail": { "href": "https://example.com/thumbnail.png", "type": "image/png", "roles": ["thumbnail"] } }
    • { "metadata": { "href": "https://example.com/metadata.json", "type": "application/json", "title": "Metadata File" } }
    • { "data": { "href": "s3://bucket/data.tif", "type": "image/tiff; application=geotiff", "description": "High-resolution GeoTIFF" } }

    Downloading Models: A Quick Guide

    Follow this guide to find, download, and integrate machine learning models from the catalog effortlessly. Ensure you check the model’s metadata for compatibility with your workflow.

    1. Search for a model: Visit the model catalog and use the search bar or filters to narrow down models that fit your requirements.
    2. Review the model details: Click on the desired model to view detailed metadata.
    3. Access the model repository: On the model's detail page, follow the provided links to the model's official repository (e.g., GitHub) to download the model files.

    STAC CORE API Documentation

    This API implements the SpatioTemporal Asset Catalog (STAC) Core API, adhering to the official STAC API Specification. Below you will find an overview of the available routes, their descriptions, and usage examples.

    Available Endpoints

    1. Root Endpoint: 
      GET /

      Returns the STAC API root with supported conformance classes and links.

    2. Conformance: 
      GET /conformance

      Returns the supported conformance classes.

    3. Collections: 
      GET /collections

      Fetches all available STAC collections.

    4. Collection by ID: 
      GET /collections/:collection_id

      Fetches a specific collection by its ID.

      • collection_id: Unique identifier of the collection
    5. Items in a Collection: 
      GET /collections/:collection_id/items

      Fetches all items within a specified collection.

    6. Item by ID: 
      GET /collections/:collection_id/items/:item_id

      Fetches a specific item within a collection by its ID.

      • collection_id: Unique identifier of the collection
      • item_id: Unique identifier of the item
    7. Search: 
      GET /search
      POST /search

      The /search endpoint allows filtering and querying STAC items based on spatial, temporal, and collection-specific criteria. It returns a STAC-compliant FeatureCollection.

      Supported Query Parameters:
      • bbox (optional): Bounding box coordinates specified as [minX, minY, maxX, maxY]. Example: ?bbox=10.0,20.0,30.0,40.0
      • datetime (optional): Temporal range in ISO 8601 format. Example: ?datetime=2023-01-01T00:00:00Z/2023-12-31T23:59:59Z
      • collections (optional): Comma-separated list of collection IDs to limit the search. Example: ?collections=collection1,collection2
      • limit (optional): The maximum number of results to return. Default: 10
      • geoFilter (optional, POST only): A GeoJSON geometry object for precise spatial filtering.

      Example Request (GET):

      GET /search?bbox=10.0,20.0,30.0,40.0&datetime=2023-01-01T00:00:00Z/2023-12-31T23:59:59Z&limit=5

      Example Request (POST):

      
      POST /search
      Content-Type: application/json
      {
        "bbox": [10.0, 20.0, 30.0, 40.0],
        "datetime": "2023-01-01T00:00:00Z/2023-12-31T23:59:59Z",
        "geoFilter": {
          "type": "Polygon",
          "coordinates": [[[10.0, 20.0], [30.0, 20.0], [30.0, 40.0], [10.0, 40.0], [10.0, 20.0]]]
        },
        "limit": 5
      }

      The response includes metadata for each item, such as geometry, properties, and collection links.

    Integrating with R

    Easily integrate machine learning models into your R workflows by following these steps:

    1. Install R: Ensure you have R installed on your system. You can download the latest version of R here.
    2. Install the required STAC client: Use the following command: 
      # Install the rstac package
install.packages("rstac")
      Then load the package: 
      # Load the rstac library
library(rstac)
    3. Connect to a STAC: Use the link to the STAC-model-data to define a STAC API-URL and create a catalog: 
      # Define the STAC API URL
url <- "LINK"
                
# Create an rstac catalog object
catalog <- stac(url)
    4. Filter and search the catalog: Query the catalog for relevant data by defining search parameters: 
      # Specify search parameters
bbox <- c(-180, -90, 180, 90)
collection <- "ml-models-rs"
                
# Perform the search
search <- stac(url) %>%
  stac_search(collections = collection,
              bbox = bbox) %>%
  get_request()
    5. Downloading data: Fetch the search results and download the assets: 
      # Retrieve items
items <- items_fetch(search)
                
# Print the results
print(items)
                
# Download assets
rstac::assets_url(items)
                
rstac::assets_download(items = items, asset_names = "water-bodies-model-pystac", output_dir = "~/Desktop/")
